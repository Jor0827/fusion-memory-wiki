# Fusion Memory Architecture

> Deep dive into the three-layer memory system design.

## Design Principles

1. **Immutable Raw Sources** — `wiki/raw/` files are never modified in place
2. **LLM-Written Pages** — `wiki/pages/` content is entirely generated by AI agents
3. **Incremental Build** — Each ingest updates all related pages, not just adds new ones
4. **Cross-Reference First** — Knowledge value comes from connections

## Layer-by-Layer

### L0: Core Index (MEMORY.md)

The single source of truth for agent identity and critical context.

```markdown
# MEMORY.md - Long-term Memory

## About
[Your agent identity and purpose]

## Key Knowledge
- Project A: [[wiki/pages/entities/project-a.md]]
- Project B: [[wiki/pages/entities/project-b.md]]

## Current Focus
[What you're working on]

## Preferences
[Your human's preferences]
```

**Purpose**: Fast context recovery on session start (~200-500 tokens).

### L1: Daily Logs (memory/YYYY-MM-DD.md)

Raw session records without synthesis.

```markdown
# 2026-04-10

## Sessions
- 02:13 — Memory system comparison (MemPalace vs LLM Wiki)
- 10:17 — MemPalace installation testing
- 10:24 — First wiki ingest

## Decisions
- Adopt fusion memory architecture (MemPalace + LLM Wiki)

## Notes
- Windows requires PYTHONIOENCODING=utf-8
```

**Purpose**: Historical trace, audit trail, recovery from corrupted L2.

### L2: Knowledge Base (wiki/)

Structured synthesis built on top of L1.

#### Entities (`wiki/pages/entities/`)

People, projects, tools, organizations.

```markdown
# [Name]

> One-line description

## Basic Info
- Type: Person/Project/Organization
- Status: Active/Paused/Completed
- Related: [[Related Entity]]

## Timeline
- YYYY-MM-DD: Event

## Key Information
[Detailed synthesized content]

## Sources
- [[Raw Source]]
```

#### Concepts (`wiki/pages/concepts/`)

Methods, theories, technologies.

```markdown
# [Concept Name]

> One-line definition

## Core Points
- Point 1
- Point 2

## Relationships
- Related Concept A: relationship
- Related Concept B: relationship

## Practical Application
[Case studies or implementation]

## Sources
- [[References]]
```

#### Decisions (`wiki/pages/decisions/`)

Why choices were made.

```markdown
# [Decision Title]

> One-line conclusion

## Background
[Why this decision was needed]

## Options
1. Option A: pros/cons
2. Option B: pros/cons

## Decision
- Conclusion: ...
- Rationale: ...
- Date: YYYY-MM-DD

## Consequences
[What changed after]

## Sources
- [[Discussion]]
```

## MemPalace Integration

### Why MemPalace?

| Requirement | MemPalace | Other Solutions |
|-------------|-----------|------------------|
| Full-text storage | ✅ | ✅ |
| Vector semantic search | ✅ (ChromaDB) | Need external DB |
| No API calls | ✅ | ❌ Most alternatives |
| Wake-up context | ✅ | ❌ |
| MCP integration | ✅ (19 tools) | Optional |

### Commands Used

```bash
# Initialize workspace
mempalace init "path/to/workspace" --yes

# Index all files
mempalace mine "path/to/workspace"

# Search semantic
mempalace search "what you're looking for"

# Get context for new session
mempalace wake-up
```

### Wake-up Output

The `wake-up` command outputs:
- L0: Agent identity + key facts (~200 tokens)
- L1: Recent session summaries (~600 tokens)
- Total: ~800 tokens of context

This can be injected into a new session for instant context.

## Operation Workflows

### Ingest (Adding New Knowledge)

1. **Save raw material** → Copy to `wiki/raw/` subdirectory
2. **Read and understand** → Extract entities, concepts, facts
3. **Update/create pages**:
   - New entity page (if new person/project)
   - Update related concept pages
   - Update decision records (if applicable)
   - Update `index.md`
4. **Cross-reference** → Add bidirectional `[[links]]`
5. **Log operation** → Append to `log.md`
6. **Re-index** → `mempalace mine "path/to/workspace"`

### Query (Answering Questions)

1. **Read index** → `wiki/index.md` to locate relevant pages
2. **Read pages** → Understand existing knowledge
3. **Read raw sources** → Fill in details if needed
4. **Synthesize answer** → Respond with citations
5. **Write back** → If valuable, create/update wiki page

### Lint (Health Check)

Periodically (weekly or during heartbeat):

- [ ] **Contradictions**: Any conflicting info across pages?
- [ ] **Orphans**: Pages with no inbound links?
- [ ] **Gaps**: Concepts mentioned but no dedicated page?
- [ ] **Stale**: Information superseded by new sources?

## Multi-Agent Scenario

### Sharing Knowledge

```
Agent A creates wiki page → Agent B reads it → Agent B builds on it
```

**Example**:
1. Agent A explores project X → creates `wiki/pages/entities/project-x.md`
2. Agent B starts session → reads wiki → knows project X without exploring
3. Agent B works on project Y → creates `wiki/pages/entities/project-y.md`
4. Agent A returns → reads wiki → knows both projects

### Conflict Resolution

When agents disagree:

1. Lint catches contradiction
2. Add both perspectives to decision page
3. Mark as "pending verification"
4. Let human user resolve

## Trade-offs

- **Storage**: MemPalace stores everything (~5MB for 400 drawers)
- **Maintenance**: Wiki requires periodic lint and updates
- **Learning curve**: Schema rules need understanding first
